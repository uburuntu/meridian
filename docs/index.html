Technology

Why it's undetectable

Censorship systems use three methods to detect proxies. Reality defeats all of them.

Deep Packet Inspection (DPI)

DPI analyzes traffic patterns to identify proxy protocols. VPNs like OpenVPN and WireGuard have distinctive packet signatures that are trivial to block.

VLESS+Reality produces traffic that is byte-for-byte identical to a normal HTTPS connection. There are no headers, no handshake patterns, no packet sizes that distinguish it from regular web browsing.

Active probing

Censors connect to suspicious servers and try to fingerprint them. If a server responds differently than the website it claims to be, it gets blocked.

Reality steals the TLS certificate from a real website (e.g., microsoft.com). When a probe connects, your server completes the TLS handshake using Microsoft's actual certificate. The probe sees a legitimate Microsoft server — because, at the TLS level, it is one. Only clients with your private key get the proxy tunnel.

TLS fingerprinting

Every TLS client (Chrome, Firefox, curl) sends a unique "Client Hello" fingerprint. Censors flag connections where the fingerprint doesn't match the claimed application.

Meridian uses uTLS to impersonate Chrome's exact TLS fingerprint — the same one used by billions of devices. Your traffic is indistinguishable from someone browsing the web with Chrome.

3

4

Architecture

How it works

Standalone mode (no domain)

HAProxy sits on port 443 and routes traffic by TLS SNI. Reality connections (SNI = microsoft.com) route to Xray, while Caddy handles everything else — serving hosted connection pages with QR codes over HTTPS, using a Let's Encrypt IP certificate (auto-renewed every 6 days).

The 3x-ui panel is reverse-proxied by Caddy on a secret path — accessible via HTTPS, no SSH tunnel needed. Every connection looks like normal HTTPS. Only clients with your private key get through to the proxy.

Domain mode (CDN fallback)

Adds three components on top of the standalone setup:

HAProxy sits on port 443 and inspects the TLS SNI header (the hostname in the Client Hello) without terminating encryption. Reality traffic (SNI = microsoft.com) routes to Xray. Domain traffic (SNI = your domain) routes to Caddy.

Caddy terminates TLS for your domain using automatic Let's Encrypt certificates and reverse-proxies the 3x-ui panel on a secret path.

VLESS+WSS inbound provides a CDN fallback path. Traffic flows through Cloudflare's CDN to your domain via WebSocket, making the connection work even if your server's IP is blocked — Cloudflare's IP ranges are too widely used to block.

For optimal stealth, use meridian scan or RealiTLScanner to find an SNI target near your server's datacenter.

Cloudflare CDN setup (domain mode)

Follow this order to avoid TLS certificate issues:

1. Add domain in Cloudflare, create A record to server IP
2. Keep cloud icon grey ("DNS only") — don't proxy yet
3. Run meridian setup — Caddy gets the TLS cert automatically
4. Switch to orange cloud (Proxied)
5. SSL/TLS → Full (Strict), Network → Enable WebSockets

Caddy obtains certs via HTTP-01 on port 80. If Cloudflare's "Always Use HTTPS" is active, it breaks the challenge. Disable it or add a page rule for /.well-known/acme-challenge/*.

Update & maintenance

Update Xray and 3x-ui:

ssh root@YOUR_SERVER cd /opt/3x-ui && docker compose pull && docker compose up -d

Or re-run meridian setup — it's safe to repeat. Manage clients with meridian client add|list|remove.

Pre-flight check

Not sure if your server is compatible? Run a pre-flight check first:

meridian check 1.2.3.4

Tests SNI target reachability, ASN match, port availability, DNS, OS, and disk space — without installing anything.

For optimal SNI stealth, scan your server's network for same-ASN targets:

meridian scan 1.2.3.4
Troubleshooting

Not connecting? Test if the server is reachable from your device:

meridian ping

Or use the web-based ping tool — no install needed, works from any device.

Connection fails immediately: Check device clock is accurate within 30 seconds. Enable "Set Automatically" in date/time settings.

Connection stops after days: Server IP may have been blocked. Switch to CDN fallback, or get a new IP and re-run.

Slow speeds: Some hosting IP ranges are throttled. Try a different provider or datacenter (Finland, Netherlands, Sweden).

Do NOT run other protocols (OpenVPN, WireGuard) on the same server — it flags the IP.

Get instant AI-powered help (copies a prompt to your clipboard):

meridian diagnostics --ai

Or collect diagnostics for a GitHub issue:

meridian diagnostics
Client management

Share access with friends — each gets their own key:

meridian client add alice

List all clients:

meridian client list

Revoke a client:

meridian client remove alice

Each client gets a unique connection key. Generates an HTML file with QR codes to share. The server IP is auto-detected from saved credentials.

Uninstall

Remove the proxy from your server:

meridian uninstall

Removes the proxy container, configs, and firewall rules. Does not remove Docker or system packages — those may be used by other applications.

Security notes

Credentials are stored at /etc/meridian/ on the server (source of truth) and cached locally at ~/.meridian/credentials/. The panel is reverse-proxied by Caddy on a secret HTTPS path in all modes — no SSH tunnel required. SSH password auth is disabled. UFW allows only ports 22 and 443.

The 3x-ui Docker image is pinned to a tested version. All secrets are redacted from terminal output.
